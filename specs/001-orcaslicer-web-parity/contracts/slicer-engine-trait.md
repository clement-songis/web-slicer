# Contrat — trait `SlicerEngine` (engine/)

Miroir 1:1 de l'API publique libslic3r consommée par la GUI —
**`audit/engine_api.json` fait foi** (constitution II). Le backend ne dépend
que de ce trait ; les implémentations sont choisies dans la composition root
(`ENGINE_IMPL=ffi|cli`) : `adapters::ffi` (v1 principal — bridge cxx lié aux
statiques Nix `libslic3r-headless`, slicing dans un process `engine-worker`
isolé, décision R1) et `adapters::cli` (fallback `orca-slicer`, même trait,
oracle de parité croisée dans les tests).

## Types miroirs (`engine::api`)

Reflètent les classes de `audit/engine_api.json` (headers listés dans
`HEADERS` de `audit/extract_engine_api.py`) :

| Type Rust | Classe C++ source | Rôle |
|---|---|---|
| `Model`, `ModelObject`, `ModelVolume`, `ModelInstance` | Model.hpp | scène : objets, pièces, instances, transformations |
| `Print`, `PrintObject`, `PrintRegion` | Print.hpp | tranchage d'un plateau |
| `SlicingParameters`, `HeightProfileSmoothingParams` | Slicing.hpp | pipeline de couches |
| `TriangleMesh`, `RepairedMeshErrors`, `TriangleMeshStats` | TriangleMesh.hpp | maillages + réparation |
| `DynamicPrintConfig` | PrintConfig.hpp | valeurs de paramètres (clés du registre `params::registry`) |
| `Preset`, `PresetBundle`, `VendorProfile` | Preset(Bundle).hpp | profils et héritage |
| `GCodeProcessorResult`, `PrintEstimatedStatistics` | GCodeProcessor.hpp | résultat/statistiques |
| `ThumbnailData`, `BuildVolume` | ThumbnailData/BuildVolume.hpp | vignettes, volume machine |

Convention : noms de méthodes Rust en snake_case de l'original
(`add_object`, `ensure_on_bed`, `estimated_print_time`…) ; signatures adaptées
aux idiomes Rust (résultats `Result<_, EngineError>`, emprunts au lieu de
pointeurs). La correspondance méthode ↔ méthode C++ est vérifiée par
`audit/check_traceability.py` (classes `used_by_gui` avec `gui_call_sites > 0`
doivent avoir un équivalent trait ou une entrée d'exclusion motivée).

## Trait (surface v1)

```rust
pub trait SlicerEngine: Send + Sync {
    // Modèle / scène
    fn load_model(&self, path: &Path, format: ModelFormat) -> Result<Model>;
    fn read_project_3mf(&self, path: &Path) -> Result<(Model, DynamicPrintConfig)>;
    fn write_project_3mf(&self, model: &Model, cfg: &DynamicPrintConfig, out: &Path) -> Result<()>;
    fn repair_mesh(&self, mesh: &TriangleMesh) -> Result<(TriangleMesh, RepairedMeshErrors)>;
    fn convert_to_mesh(&self, path: &Path) -> Result<TriangleMesh>; // STEP → mesh (R7)

    // Opérations de scène déléguées au moteur
    fn arrange(&self, model: &mut Model, bed: &BuildVolume, cfg: &ArrangeParams) -> Result<()>;
    fn orient(&self, object: &mut ModelObject) -> Result<()>;

    // Presets
    fn resolve_preset_chain(&self, chain: &[RawPreset]) -> Result<DynamicPrintConfig>;
    fn validate_config(&self, cfg: &DynamicPrintConfig) -> Result<Vec<ConfigWarning>>;

    // Tranchage (unité = plateau)
    fn slice(&self, req: SliceRequest, progress: ProgressSink, cancel: CancelToken)
        -> Result<SliceResult>; // SliceResult: gcode path, GCodeProcessorResult, thumbnails

    // Prévisualisation
    fn parse_gcode(&self, path: &Path) -> Result<GcodePreview>; // couches/segments/stats (R6)
}
```

`ProgressSink` : callback `fn(phase: &str, ratio: f32)` relayé vers la file ;
`CancelToken` : annulation coopérative (kill du process CLI).

## Garanties d'implémentation (testées)

1. **Parité G-code** (SC-003) : pour un corpus de référence (10 modèles ×
   5 combinaisons de presets, figé dans `engine/tests/fixtures/`), le G-code
   produit est identique à celui d'`orca-slicer` desktop aux métadonnées près
   (lignes `; generated by` etc. normalisées avant diff).
2. **Erreurs restituées** : tout échec moteur devient `EngineError` structurée
   { code, message d'origine, objet/paramètre concerné si identifiable } (FR-032).
3. **Isolation** : aucune écriture hors du répertoire de travail fourni.
4. **Substituabilité** : la suite de tests du trait est générique
   (`fn test_engine(e: &dyn SlicerEngine)`) et passe telle quelle sur les
   implémentations FFI **et** CLI (gate de sortie P1), puis sur la future
   implémentation Rust native (constitution II).
5. **Parité croisée FFI/CLI** : sur le corpus de référence, FFI et CLI
   produisent le même G-code (mêmes réglages → même cœur libslic3r) ; tout
   écart est un bug de bridge.
